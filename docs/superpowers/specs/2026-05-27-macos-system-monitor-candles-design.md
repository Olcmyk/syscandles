# macOS系统监控K线图应用设计文档

## 项目概述

开发一个macOS原生应用，将系统实时数据（CPU占用、GPU占用、RAM、电量、上传网速、下载网速）以K线图形式展示，类似金融交易终端（如Wind终端）的体验。

## 技术栈

- **前端：** Vanilla JavaScript + LightweightCharts
- **后端：** Rust + Tauri + sysinfo
- **数据存储：** SQLite
- **打包格式：** macOS DMG

## 架构设计

### 整体结构

**前端（Vanilla JS + LightweightCharts）：**
- 单页应用布局
- 左侧：指标列表（6个系统指标）
- 右侧：K线图表区域
- 顶部：工具栏（时间周期切换：5秒、1分钟、5分钟、15分钟、1小时）

**后端（Rust + Tauri + sysinfo）：**
- 系统监控服务：定期采集系统数据
- 数据存储层：SQLite本地数据库存储历史数据
- Tauri命令接口：提供前端调用的API
- 事件推送：实时数据推送到前端

**打包：**
- Tauri打包为macOS DMG安装包

### 数据采集策略

- 后台服务以**5秒为基础频率**采集原始数据点
- 前端请求不同周期的K线时，后端从原始数据聚合计算
- 应用关闭时停止采集，重新打开时从上次时间点继续
- **允许数据缺失**（用户关机期间不采集）

### 数据量估算

- 每条数据约50-100字节（时间戳+6个指标值）
- 1小时 = 720条 ≈ 72KB
- 1天 = 17,280条 ≈ 1.7MB
- 1个月 ≈ 50MB
- 1年 ≈ 600MB

## 组件设计

### 前端组件

**1. IndicatorList（指标列表）**
- 位置：左侧边栏
- 功能：显示6个指标按钮（CPU、GPU、RAM、电量、上传速度、下载速度）
- 交互：点击切换当前显示的图表，高亮当前选中项

**2. ChartView（图表视图）**
- 位置：右侧主区域
- 功能：使用LightweightCharts渲染K线图
- K线数据：开高低收四个值（5秒内的起始值、最高值、最低值、结束值）

**3. Toolbar（工具栏）**
- 位置：顶部
- 功能：
  - 时间周期切换按钮（5秒、1分钟、5分钟、15分钟、1小时）
  - 日期范围选择器（用于跳转到指定日期查看历史K线数据，默认显示最近1小时）

**4. RealtimeUpdater（实时更新器）**
- 功能：监听Tauri事件，接收后端推送的最新数据点
- 行为：更新当前显示的图表

### 后端组件（Rust）

**1. DataCollector（数据采集器）**
- 功能：使用sysinfo库每5秒采集一次系统数据
- 采集指标：
  - CPU使用率（0-100%）
  - GPU使用率（0-100%）
  - 内存使用率（0-100%）
  - 电量百分比（0-100%）
  - 上传速度（KB/s）
  - 下载速度（KB/s）
- 行为：采集后写入数据库

**2. DatabaseManager（数据库管理器）**
- 功能：SQLite操作封装
- 接口：
  - 插入原始数据
  - 查询历史数据
  - 聚合K线数据

**3. TauriCommands（命令接口）**
- 暴露给前端的命令：
  - `get_historical_data(indicator, period, start_time, end_time)` - 获取历史K线数据
  - `get_latest_data()` - 获取最新一条原始数据
  - `start_collection()` - 启动采集服务
  - `stop_collection()` - 停止采集服务

**4. EventEmitter（事件发射器）**
- 功能：每次采集到新数据后，通过Tauri事件推送给前端
- 事件名：`data-update`

## 数据流设计

### 启动流程

1. 应用启动 → Tauri初始化 → 检查SQLite数据库是否存在，不存在则创建
2. 前端加载 → 调用`start_collection()`启动后台采集服务
3. DataCollector启动定时器，每5秒采集一次数据并写入数据库
4. 前端默认显示CPU指标，调用`get_historical_data()`加载最近1小时的1分钟K线数据
5. EventEmitter开始推送实时数据，前端接收后更新图表

### 用户切换指标

1. 用户点击左侧指标列表（如GPU）
2. 前端调用`get_historical_data('gpu', current_period, ...)`获取该指标的历史数据
3. 图表切换显示GPU的K线
4. 实时更新器继续接收数据，但只更新当前显示的指标图表

### 用户切换时间周期

1. 用户点击工具栏周期按钮（如从1分钟切换到5分钟）
2. 前端调用`get_historical_data(current_indicator, '5min', ...)`
3. 后端从原始5秒数据聚合成5分钟K线返回
4. 图表重新渲染新周期的K线

### 数据聚合逻辑（后端）

- 原始数据表存储每5秒的采集点：`(timestamp, cpu, gpu, ram, battery, upload, download)`
- 聚合K线时，按时间窗口分组：
  - **开盘值** = 窗口内第一条数据
  - **最高值** = 窗口内最大值
  - **最低值** = 窗口内最小值
  - **收盘值** = 窗口内最后一条数据
- 如果某个时间窗口没有数据（电脑关机），该K线缺失，图表显示断点

## 数据存储设计

### SQLite数据库结构

```sql
-- 原始数据表
CREATE TABLE raw_data (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp INTEGER NOT NULL,  -- Unix时间戳（秒）
    cpu REAL NOT NULL,           -- CPU使用率 0-100
    gpu REAL NOT NULL,           -- GPU使用率 0-100
    ram REAL NOT NULL,           -- 内存使用率 0-100
    battery REAL NOT NULL,       -- 电量百分比 0-100
    upload REAL NOT NULL,        -- 上传速度 KB/s
    download REAL NOT NULL       -- 下载速度 KB/s
);

CREATE INDEX idx_timestamp ON raw_data(timestamp);
```

### 数据保留策略

- 保留所有历史数据，不自动删除
- MVP阶段不提供数据清理功能，未来可在设置中添加删除N天前数据的选项
- 预估存储：1年约600MB，对现代硬盘来说可以接受

### 数据库位置

- macOS: `~/Library/Application Support/com.macoscandles.app/data.db`
- 使用Tauri的`app_data_dir()`获取标准应用数据目录

### 缺失数据处理

- 电脑关机期间不采集数据，时间轴上会有空白
- 前端图表显示时，LightweightCharts会自动处理时间序列的断点
- 不进行数据插值或填充，保持真实性

## 错误处理

### 系统数据采集失败

- 如果sysinfo无法获取某个指标（如某些Mac不支持GPU监控），该指标值记录为-1或NaN
- 前端显示该指标时，如果数据全是无效值，显示"该指标不可用"提示
- 部分指标失败不影响其他指标的采集

### 数据库操作失败

- 写入失败时记录到日志，跳过本次采集，不中断采集服务
- 读取失败时向前端返回错误，前端显示"加载数据失败"提示
- 数据库文件损坏时，尝试备份并重建空数据库

### 前端渲染错误

- 如果K线数据为空（新安装或查询范围无数据），显示"暂无数据"占位符
- 如果LightweightCharts初始化失败，降级显示简单的文本数据列表

### 网络速度采集特殊处理

- macOS上网络速度需要计算两次采集之间的流量差值
- 第一次采集时无法计算速度，记录为0
- 如果网络接口变化（如切换WiFi），可能出现异常值，需要做合理性检查（如速度>1GB/s视为异常，记录为0）

## 测试策略

### 后端测试

**单元测试：**
- 测试数据聚合逻辑（5秒数据聚合成不同周期K线的正确性）
- 测试数据库CRUD操作

**集成测试：**
- 测试DataCollector能否正常采集数据
- 测试EventEmitter能否正常推送事件

**手动测试：**
- 在真实macOS环境验证sysinfo能否获取所有6个指标

### 前端测试

**手动测试为主：**
- 启动dev server，验证图表切换、周期切换、实时更新是否正常

**边界情况测试：**
- 空数据场景
- 单条数据场景
- 大量历史数据的渲染性能

### 端到端测试

- 启动应用后持续运行1小时，验证数据采集稳定性
- 关闭应用再重启，验证数据持久化和断点恢复
- 切换所有指标和所有时间周期，验证聚合计算正确性

### 打包测试

- 在干净的macOS系统上安装DMG，验证首次启动、权限请求、数据目录创建
- 验证应用签名和公证（如果需要分发）

## 技术风险和缓解

### 风险1：sysinfo在macOS上的GPU监控支持
- **风险：** sysinfo可能无法获取GPU使用率
- **缓解：** 如果sysinfo不支持，考虑使用macOS系统命令（如`ioreg`）或标记为"不可用"

### 风险2：LightweightCharts性能
- **风险：** 大量历史数据可能导致渲染性能问题
- **缓解：** 限制单次加载的数据点数量（如最多显示1000根K线），提供分页或懒加载

### 风险3：后台采集对系统性能的影响
- **风险：** 5秒频率的采集可能影响系统性能
- **缓解：** 监控采集服务本身的资源占用，如果过高则调整采集频率或优化代码
